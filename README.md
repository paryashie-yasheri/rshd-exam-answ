## Shared buffers
Разделяемое буферное пространство
- Хранит копии блоков данных, считанных из файлов данных
- Служит для минимизации числа операций обмена с диском
- Если нужного блока/страницы данных нет в SB, то он читается с диска и помещается в SB
- Совместно используется всеми фоновыми и пользовательскими процессами экземпляра
- Можно настроить через `shared_buffers`
- По умолчанию 128 МБ

## WAL buffers
- WAL -- Write Ahead Log
- Хранит информацию об изменениях данных в БД — записи XLOG
- Изменениям присваивается LSN
	- Log sequence number
- Эта информация используется для воссоздания актуального состояния данных в случае восстановления БД
- Настраивается через параметр `wal_buffers`

## Tablespace
- Позволяют задать пути (директорию в файловой системе), где будут храниться объекты БД (вне PGDATA)
- Позволяют управлять физическим расположением объектов БД

### Стандартные табличные пространства
- `pg_default` — соответствуют директории base в PGDATA
	- По умолчанию здесь хранятся файлы данных, соответствующие объектам БД
- `pg_global` — соответствуют директории global в PGDATA
	- `pg_database`, `pg_authid`, `pg_tablespace`, некоторые другие каталоги и индексы

## CLOG buffers
- CLOG -- Commit Log
- Хранит данные о статусе проведения транзакций (e.g. "в процессе", "закончена", etc)
- Размер автоматически устанавливается СУБД
- Доступен серверным процессам

# Фоновые процессы в PostgreSQL

![Фоновые процессы PostgreSQL](images/Pasted%20image%2020260618221419.png)

## Процесс записи в БД (writer process / background writer)

- Переодически записывает измененные (заполненные, грязные) страницы из SB на диск в файлы данных
- Помечает записанные страницы чистыми
- Облегчает работу процесса checkpointer
- bgwriter_delay

## Работа с контрольными точками / checkpointer

- Контрольная точка — точки в последовательности транзакций, в которые произведена синхронизация результатов выполненных операций с файлами на диске
-  Создание контрольной точки:
	1) WAL-буфер синхронизируется с диском
	2) Грязные страницы записываются на диск
	3) Контрольная точка фиксируется в логах
- Контрольная точка создается, когда зполнен max_wal_size или через время checkpoint_timeout (по умолчанию — 300 секунд) — в зависимости от того, что будет раньше

## Loggin collector

- Записывает сообщения, отправленные в stderr, в лог-файлы
- Для работы нужно, чтобы был установлен параметр logging_collector

## Archiver
- Копирует созданные WAL-файлы в указанное место
- По умолчанию выключен

## Stats collector

- Служит для сбора различных статистических данных о БД
- Статистика хранится в промежуточных файлах и через них используется другими процессами
- Директория с временными файлами определяется параметром stats_temp_directory

```
pg_stat_*
```

# Файловая структура и конфигурация PostgreSQL

## Файловая структура

- PGDATA — путь к директории данных кластера; установлена заранее
	- global — содеожит таблицы уровня кластера (pg_database)
	- pg_wal — директория с WAL-файлами
	- pg_xact — директория с данными о коммитах транзакций (CLOG)
	- base — файлы данных базы данных кластера
- Объектам БД соответствуют файлы данных:
	- если до 1Gb (по умолчанию) — объекту соответствует 1 файл данных
	- если >1Gb — объекту соответствуют файлы-сегменты
	- Находятся в $PGDATA/base в директории соответствующей бд

- По умолчанию располагается в PGDATA

	- search_path = 'someSchema, public'
	- shared buffers = 128MB
- Определяют значение для всего кластера

## General
- Для предотвращения ситуаций, связанных с появлением несогласованных данных можно использовать **журнал**/**лог**
- СУБД сохраняет информацию об изменениях и фиксациях данных в журнале:
	- Первоначально запись сохраняется в соответствующем буфере (буфере журнала)
	- Буфер журнала синхронизируется с файлами журнала на диске (WriteLog)

## UNDO/REDO Logs

### UNDO Log
- Журнал отмены — это один из вариантов ведения журналов
- Возможные записи в журнале отмены:

| Команда                       | Описание                                                      |
| ----------------------------- | ------------------------------------------------------------- |
| BEGIN Tx                      | Начало транзакции                                             |
| Tx: R, old_val                | Изменение:<br>Транзакция Tx переписывает значение old_var в R |
| COMMIT Tx                     | Успешное завершение транзакции                                |
| ABORT Tx                      | Преждевременное завершение транзакции                         |
| START CHKn: $T_n, \dots, T_m$ | Начало создания контрольной точки                             |
| END CHKn                      | Завершение создания контрольной точк                          |

#### Правила записи в UNDO LOG
- Запись изменения данных (Tx: R, old_val) записывается в журнал (на диск) ДО сохранения обновленного значения на диске
- `COMMIT Tx` должен быть добавлен в журнал **ПОСЛЕ** обновления всех файлов данных, связанных с изменениями данной транзакции (после синхронизации буферного пространства данных с файлами данных)

#### Использование журнала отмены для восстановления данных
1) СУБД сканирует журнал от новых записей до старых. Отдельно фиксируются транзакции:
	- Группа 1:
		- Для которых есть запись COMMIT Tx
	- Группа 2:
		- Транзакции, для которых есть запись ABORT Tx, незавершенные транзакции
2) Когда в журнале встречается запись изменения (Tx: R, old_val), анализируется Tx. Если Tx:
	- Из группы 1:
		- Действия не предпринимаются
	- Из группы 2: 
		- Для R восстанавливается old_var (происходит отмена изменений)
3) СУБД записывает ABORT Tx для каждой незавершенной транзакции из группы 2
4) Запускается операция WriteLog, чтобы записи вида ABORT Tx оказались в журнале

### Контрольные точки

#### Создание контрольной точки
1. В журнал добавляется START CHKn: $T_{n}, \dots, T_{m}$
2. Ожидание завершения транзакций, которые работали в момент создания контрольной точки($T_{n}, \dots, T_{m}$)
	- В это время могут начинаться другие транзакции
3. В журнал добавляется END CHKn, фиксируется на диске через WriteLog
4. Записи перед START CHKn, у которой есть END CHKn, могут быть удалены

## Steal/No-steal & Force/No-force

| Метод        | Описание                                                                                                                                                 | Используется в PostgreSQL |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------- |
| **no-steal** | — подход, при котором страница из буфера данных не может быть записана на диск до завершения транзакции, в ходе которой она была изменена                | ❌                         |
| **steal**    | — если позволено записывать в файлы данных данные еще не завершенных транзкций (для оптимизации запросов к диску или если буферная память заканчивается) | ✅                         |
| **force**    | — если требуется, чтобы все страницы, измененные транзакцией, записывались сразу перед COMMIT                                                            | ❌                         |
| **no-force** | — изменения, осуществленные завершенными транзакциями могут быть не зафиксированы на диске                                                               | ✅                         |

## ARIES

### -- Algorithms for Recovery and Isolation Expoiting Semantics a.k.a. Алгоритм  для восстановления данных

### Базируется на:
- WAL
- steal
- no-force

### Этапы
1) Анализ
	- Поиск "грязных" страниц и незавершенных транзакций
2) REDO
	- Повторение всех действий
3) UNDO
	- Откат действий незавершенной транзакции

> NB: Для работы требуется LSN, таблица транзакций, таблица "грязных" страниц
